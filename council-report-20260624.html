Là où le council converge

  • 5/5 → Option A (AMAPI) pour le MVP solo.
  • OTA système : critère à rayer — impossible sans clés OEM dans A comme dans B. 3 des 5 actions ont la même limite partout.
  • Le DPC custom n'est pas "plus open", juste plus de surface à maintenir seul.
  • Q2 validée : "contrat d'API d'abord" n'est pas une étape — c'est le produit lui-même.
  • Q4 défendable à condition de nommer la frontière : "control plane ouvert, moteur Google".

Là où le council s'oppose

Le désaccord structurant porte sur le lock-in (Q3) — et les deux camps ont raison à des stades différents.

Pas de lock-in (FP, EXP, EXE)

Raisonnement au niveau code : un domaine à verbes métier rend le DPC custom branchable plus tard. Réversible, gratuit.

Lock-in réel (CON)

Raisonnement au niveau flotte déployée : migrer AMAPI→DPC = factory reset de chaque appareil terrain. L'abstraction logicielle ne sauve rien.

Arbitrage : un MVP solo au stade zéro n'a pas de flotte → le lock-in d'enrollment est un risque go-to-market futur, pas MVP.

Angles morts détectés au peer review

Aucun des 5 advisors ne les avait vus seul.

  • Éligibilité administrative AMAPI (4 reviewers) : Enterprise validée Google, Managed Google Play / Cloud Identity, quotas. L'obstacle n'est pas le code, c'est l'approbation — peut casser le "lundi matin".
  • Émulateur ≠ terrain : LOCK_TASK, reboot et silent install dépendent du GMS/OEM réels. Il manque au moins un appareil physique.
  • RGPD / souveraineté : les données de flotte résident chez Google — frontal avec un positionnement "open/souverain".
  • Wrapper sans moat : Google peut déprécier ou tarifer l'API.
  • L'acheteur réel : qui paie, et pourquoi pas Scalefusion/Esper ? Positionnement à trancher avant la techno.

La recommandation

Option A (AMAPI), traitée comme C dès le jour 1 — au niveau du vocabulaire, pas du code. Un seul backend, des verbes métier neutres dans le contrat (reboot, enterKiosk, installApp), un ProviderInterface nommé mais à implémentation unique. N'écrivez pas l'abstraction tant qu'AMAPI est seul.

Séquencement : le control plane (intention → policy → état réel) est la thèse ; le moteur d'exécution est interchangeable. Construisez dans cet ordre.

Lock-in : aucun au stade MVP. Le seul vrai verrou est l'enrollment en production → décision future, pas MVP.

Deux conditions non négociables : (1) vérifier l'éligibilité AMAPI avant tout code ; (2) nommer la frontière open dès le pitch, et acheter un appareil Android physique.

Le premier pas à faire

Avant toute ligne de code : créez le projet GCP, activez l'Android Management API, et tentez de créer une Enterprise de test pour découvrir maintenant si l'éligibilité administrative (validation Google / Managed Google Play / quotas) est un mur ou une formalité. C'est le seul fait qui peut invalider toute la stratégie — et qu'aucun advisor n'avait vérifié.
