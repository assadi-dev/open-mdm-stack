02Côté serveur API — plan d'enrôlement

Distinction fondamentale · statique vs dynamique

Partie Nature Source
Component name du DeviceAdminReceiverStatiqueConfig
URL de download de l'APKStatiqueConfig
Checksum de signature de l'APKStatiqueConfig · keystore release
ADMIN_EXTRAS_BUNDLE → token + URL + tenantDynamiquePar enrôlement
WiFi SSID / password (optionnel)DynamiquePar enrôlement

Un endpoint assemble [config statique] + [token frais] → JSON QR.

Modèle de données (Drizzle)

  • enrollment_tokens — id, token (haute entropie), tenant_id, created_by, expires_at, max_uses, used_count, status (active/revoked/exhausted), opt. policy_id/group_id, opt. device_name_template.
  • devices — créé à l'enrôlement : id, tenant_id, enrollment_token_id, serial, model, android_version, device_jwt_id, status, last_seen, enrolled_at.

Endpoints

Endpoint Auth Rôle
POST /enrollment-tokensusercrée un token + renvoie payload QR + image
GET /provisioning/dpc.apkpublique / cookie-headersert l'APK pendant le provisioning
POST /enrolltoken devicevalide le token, crée le device, renvoie le JWT
POST /devices/:id/deenrolluserrévoque le device + son JWT
Piège

Le download APK. À l'étape 4 le device n'est pas encore enrôlé → aucun credential. L'endpoint APK doit être accessible sans auth device : protège-le avec le PROVISIONING_PACKAGE_DOWNLOAD_COOKIE_HEADER (header secret dans le QR) plutôt que de l'exposer totalement.

Lifecycle du token

Single-use (ou max_uses limité) + TTL court + scopé tenant. À l'étape 9 le serveur incrémente used_count et refuse si expiré / révoqué / épuisé. Bloque le replay.

03Lib de génération QR

Côté serveur · qrcode (node-qrcode). Bon défaut :

  • mature, maintenu, zéro dépendance lourde ;
  • sort en data URL / PNG / SVG / terminal ;
  • niveaux de correction d'erreur réglables.
Note

Taille du payload. Download URL + checksum + bundle rendent le QR dense. Utiliser un niveau de correction L ou M (pas H) sinon le QR devient illisible / trop gros. node-qrcode auto-dimensionne la version — à tester sur device réel.

Alternative streaming : qr-image (moins maintenu) — non recommandé.

Côté Android · aucune lib QR nécessaire pour le provisioning Device Owner (le Setup Wizard scanne). Un scanner (ML Kit Barcode / ZXing) ne servirait qu'à un autre mode d'enrôlement in-app — hors scope.

04Côté Android — plan du provisioning

le récepteur

AndroidManifest

  • DeviceAdminReceiver avec permission BIND_DEVICE_ADMIN, intent-filters : DEVICE_ADMIN_ENABLED, PROFILE_PROVISIONING_COMPLETE.
  • Ressource XML device_admin listant les policies utilisées.

Le receiver

  • onProfileProvisioningComplete(intent) → moment clé : récupérer l'ADMIN_EXTRAS_BUNDLE → token + URL serveur + tenant.
  • onEnabled → admin actif.
Android 10+
Obligatoire, sinon le provisioning échoue. Fournir deux activités :
  • ACTION_GET_PROVISIONING_MODE → renvoie le mode (fully-managed / device owner) ;
  • ACTION_ADMIN_POLICY_COMPLIANCE → finalise la conformité après provisioning (déclenche /enroll).
Le détail qui fait « marche sur émulateur ancien » mais « échoue sur device récent ».

Séquence post-provisioning

  1. Lire le bundle (token + URL).
  2. Collecter les infos device (serial via DevicePolicyManager, model, version).
  3. POST /enroll (Retrofit) → recevoir le JWT device.
  4. Stocker le JWT (EncryptedSharedPreferences / Keystore).
  5. Appliquer les policies, démarrer le foreground service + ouvrir le stream SSE.
  6. Marquer la conformité (fin de ADMIN_POLICY_COMPLIANCE).

05Sécurité

  • Token — haute entropie, single-use, TTL court, scopé tenant → anti-replay.
  • Checksum de signature dans le QR = protection MITM : le device refuse un APK substitué. Calculé depuis le keystore release (SHA-256 du certificat, base64 url-safe).
  • HTTPS obligatoire — l'URL serveur dans le bundle doit être https.
  • JWT device — signé, porte device_id + tenant_id, révocable (table de révocation / rotation) pour le de-enroll.
  • Endpoint APK — cookie-header secret plutôt que public.
