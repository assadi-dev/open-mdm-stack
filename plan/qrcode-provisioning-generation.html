Génération du QR de provisioning

Données d'entrée · étapes · payload · schéma

EndpointPOST /api/v1/enrollment-tokens — auth user (dashboard).
SortieJSON de provisioning Android Device Owner + image QR (data URL / PNG / SVG).
Libqrcode (node-qrcode) — correction d'erreur M (payload dense).

01Schéma de génération

Dashboard → API → QR

Rappel. Le device ne dialogue pas ici — le Setup Wizard natif d'Android scannera le QR plus tard. La génération est 100 % côté serveur.

02Données d'entrée nécessaires

Ce qu'il faut réunir avant de générer

Dynamique — par enrôlement, depuis le body de la requête

Champ Requis Note

max_usesdéfaut 1single-use recommandé

Le enrollment_token n'est pas une entrée : il est généré par le serveur (haute entropie) à l'étape 1.

◆ MVP MONO-TENANT Ce MVP est mono-tenant (1 serveur = 1 organisation) : pas de tenant_id dans le payload ni le modèle de données. Le scope multi-tenant pourra être réintroduit plus tard sans casser le contrat d'enrôlement.

03Étapes de génération (API)

Du body de la requête à l'image renvoyée

  1. 1 Auth + validation (Zod). Vérifie l'utilisateur dashboard, parse le body : ttl, max_uses, options WiFi/policy.
  2. 2 Génère le token. crypto.randomBytes → haute entropie ; calcule expires_at ; persiste la ligne enrollment_tokens (status=active, used_count=0).
  3. 3 Charge la config statique. Component name, URL APK, checksum signature, cookie-header — depuis l'ENV.
  4. 4 Assemble ADMIN_EXTRAS_BUNDLE. { enrollment_token, server_url } — la partie lue par le DPC après provisioning.
  5. 5 Construit le payload JSON. Fusionne config statique + bundle dans l'objet aux clés android.app.extra.* (voir §04).
  6. 6 Sérialise + encode QR. JSON.stringify → QRCode.toDataURL(payload, { errorCorrectionLevel: 'M' }). Sortie data URL, PNG ou SVG.
  7. 7 Répond au dashboard. { token_id, expires_at, qr } — le dashboard affiche l'image, prête à être scannée.

04Structure du payload

Clés officielles Android Device Owner — encodées dans le QR

{
  // statique — config serveur
  "android.app.extra.PROVISIONING_DEVICE_ADMIN_COMPONENT_NAME": "com.openmdm.dpc/.MdmDeviceAdminReceiver",
  "android.app.extra.PROVISIONING_DEVICE_ADMIN_PACKAGE_DOWNLOAD_LOCATION": "https://mdm.example.com/api/v1/provisioning/dpc.apk",
  "android.app.extra.PROVISIONING_DEVICE_ADMIN_SIGNATURE_CHECKSUM": "gJD2YwtOiWJHkSMkkIfLRlj-quNqG1fb6v100QmzM9w=",
  "android.app.extra.PROVISIONING_DEVICE_ADMIN_PACKAGE_DOWNLOAD_COOKIE_HEADER": "X-Provisioning-Token: <secret>",

  // dynamique — lu par le DPC après provisioning
  "android.app.extra.PROVISIONING_ADMIN_EXTRAS_BUNDLE": {
    "enrollment_token": "9f2c8a…haute-entropie",
    "server_url": "https://mdm.example.com"
  },

  // optionnel — réseau de provisioning
  "android.app.extra.PROVISIONING_WIFI_SSID": "OfficeWiFi",
  "android.app.extra.PROVISIONING_WIFI_PASSWORD": "••••••••",
  "android.app.extra.PROVISIONING_WIFI_SECURITY_TYPE": "WPA",
  "android.app.extra.PROVISIONING_SKIP_ENCRYPTION": false
}

Seul ADMIN_EXTRAS_BUNDLE change d'un enrôlement à l'autre. Le reste est figé par la config.

⚠ DENSITÉ Avec URL + checksum + bundle, le QR est dense. Utiliser la correction d'erreur L ou M, jamais H (sinon QR illisible / trop gros). node-qrcode auto-dimensionne la version — à tester sur device réel.
↪ CODE ACTUEL L'implémentation de départ (features/qrcode/service.ts) génère depuis un simple text en errorCorrectionLevel: 'H' et écrit un fichier. À faire évoluer : payload provisioning structuré + niveau M + sortie data URL.
