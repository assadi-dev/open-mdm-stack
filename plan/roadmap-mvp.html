01Principe d'ordonnancement

Dé-risquer d'abord le maillon incertain — le provisioning Device Owner sur device physique. Tant qu'un DPC ne se provisionne pas sur le matériel cible, le backend élégant reste théorique. On remonte ensuite la chaîne : auth admin → tokens + QR → boucle /enroll → cycle de vie.

P1 (M0)  Spike DPC ----------------+  prouve le hardware
P2       Auth admin ---------------+  socle d'identite
P3       Tokens + QR serveur ------+  generation legitime
P4       Boucle /enroll + devices -+  device <-> serveur reel
P5       Dashboard enrolement -----+  rend utilisable
P6       Cycle de vie + revocation +  hygiene avant les commandes

02Les phases

P1Spike DPC Device Owner

Dé-risquage · priorité absolue

But — réussir UN provisioning Device Owner sur un device physique réel.

  • apps/android : DPC Kotlin minimal — DeviceAdminReceiver (BIND_DEVICE_ADMIN, intents DEVICE_ADMIN_ENABLED + PROFILE_PROVISIONING_COMPLETE), XML device_admin.
  • Activités Android 10+ obligatoires : ACTION_GET_PROVISIONING_MODE + ACTION_ADMIN_POLICY_COMPLIANCE (sinon le provisioning ne se termine pas sur device récent).
  • Keystore release → checksum SHA-256 du certificat (base64 url-safe).
  • QR de provisioning hardcodé (composant + URL APK + checksum + ADMIN_EXTRAS_BUNDLE factice).
  • onProfileProvisioningComplete lit le bundle et logue token + URL (pas d'appel réseau encore).
DoD  factory-reset → scan QR → DPC en Device Owner → bundle lu et loggé. Dépend  aucune.

P2Authentification admin

Parallèle à P1

But — un admin se connecte et obtient une session / JWT pour garder les endpoints métier.

  • Activer email/password dans auth.ts (Better Auth) ; corriger trustedOrigins (actuellement = BETTER_AUTH_SECRET, bug).
  • Middleware requireUser → protéger les futures routes dashboard.
  • Provisionnement du 1er admin : seed ou CLI (pas de self-signup ouvert sur un MDM).
  • apps/web : page login + garde des pages dashboard.
DoD  login web OK, session persistée, endpoint protégé → 401 sans / 200 avec. Dépend  aucune.

P3Tokens d'enrôlement + génération QR serveur

Après P2

But — produire un QR de provisioning légitime et révocable.

  • Schéma enrollment_tokens : id, token (haute entropie), created_by, expires_at, max_uses, used_count, status, opt. policy_id / device_name_template. (sans tenant_id)
  • POST /enrollment-tokens (auth user) → assemble [config statique] + [token] → payload JSON → QR (correction L/M, pas H).
  • GET /provisioning/dpc.apk → sert l'APK avant enrôlement, protégé par header secret (PROVISIONING_PACKAGE_DOWNLOAD_COOKIE_HEADER), pas public.
DoD  un admin génère un QR ; un device de P1 se provisionne dessus. Dépend  P2 · profite de P1.

P4Boucle d'enregistrement /enroll + modèle devices

Après P1 + P3

But — fermer la chaîne device ↔ serveur.

  • Schéma devices : id, enrollment_token_id, serial, model, android_version, device_jwt_id, status, last_seen, enrolled_at. (sans tenant_id)
  • POST /enroll (auth = token device) : valide le token, incrémente used_count, crée le device, émet un JWT device révocable.
  • DPC (suite de P1) : collecte serial/model/version → POST /enroll (Retrofit) → stocke le JWT en EncryptedSharedPreferences / Keystore → marque la conformité.
DoD  device reset → provisioning → apparaît enrolled avec son JWT, sans étape manuelle. Dépend  P1 + P3.

P5Dashboard d'enrôlement

Après P2·P3·P4

But — rendre l'enrôlement utilisable sans curl.

  • apps/web : écran « Enrôler un appareil » → génère / affiche le QR (data URL / SVG), TTL et usage du token visibles.
  • Liste des appareils enrôlés (statut, last_seen, modèle).
DoD  un admin enrôle un device de bout en bout depuis l'UI. Dépend  P2, P3, P4.

P6Cycle de vie & révocation

Après P4

But — hygiène avant d'ajouter les commandes à distance.

  • POST /devices/:id/deenroll (auth user) → révoque le device + invalide son JWT (table de révocation / rotation).
  • Révocation d'enrollment_tokens (status revoked).
  • Côté DPC : gérer un JWT révoqué (relock / wipe selon politique — à cadrer).
DoD  de-enroll coupe l'accès du device (401 sur appels suivants). Dépend  P4.

03Hors scope MVP sans se fermer la porte

  • Distribution de policies / APK, file de commandes, SSE (GET /devices/:id/stream) + heartbeat + polling — pilier « commandes » suivant.
  • Multi-tenant (réintroductible si le contrat d'enrôlement reste neutre).
  • OTA système (impossible sans clés OEM / root — un DO ne fait que setSystemUpdatePolicy).

04Sécurité transverse à tenir dès P3/P4

  • Token — haute entropie, single-use / max_uses, TTL court → anti-replay.
  • Checksum de signature dans le QR → anti-MITM sur l'APK.
  • HTTPS obligatoire pour l'URL serveur du bundle.
  • JWT device révocable ; endpoint APK derrière header secret, pas public.

05Chemin critique

P1 d'abord

P1 (spike DPC) débloque tout le reste — à lancer en premier et en parallèle de P2 (auth). P3 → P4 → P5 sont séquentiels. P6 ferme la boucle avant le pilier « commandes ».
